\section{Requisitos}

El usuario necesita:

\begin{itemize}
    \item Cliente HTTP (navegador, Postman o curl).
    \item Conexión a Internet.
    \item Token de autenticación Bearer para la bandeja TI, el centro operativo, la demostración y las operaciones protegidas de OpenAPI. El portal y la creación de solicitudes no requieren que el usuario introduzca el token.
\end{itemize}

\section{Uso por perfiles}

El prototipo contempla varios perfiles de uso:

\begin{itemize}
    \item \textbf{Usuario interno}: utiliza el portal web para registrar solicitudes de acceso, entorno, aplicación, configuración o incidencia.
    \item \textbf{Equipo TI}: utiliza la bandeja protegida para consultar, filtrar, asignar, priorizar y cerrar solicitudes.
    \item \textbf{Sistema externo}: puede generar solicitudes mediante la Logic App sin conocer los detalles internos de la API.
    \item \textbf{Administrador cloud}: mantiene configuración, secretos, despliegue y monitorización en Azure.
\end{itemize}
\section{Operaciones disponibles}

\subsection{Consultar estado del servicio}

\begin{verbatim}
GET /health
\end{verbatim}

La respuesta indica si el servicio está operativo y si utiliza almacenamiento local o Azure Blob Storage. Esta operación es pública para facilitar la comprobación de disponibilidad.

\subsection{Crear solicitud desde el portal}

El usuario puede abrir el portal web, seleccionar el tipo de solicitud, el servicio empresarial, el activo afectado y el entorno, introducir título, descripción y correo del solicitante, y enviar el formulario. La API devuelve un identificador, prioridad, clasificación, impacto y recomendación operativa. El catálogo evita escribir nombres libres y permite relacionar cada petición con un responsable técnico.

Cuando el servicio y el entorno tienen una criticidad elevada, la respuesta informa del impacto calculado y del aprobador responsable, como se observa en la figura \ref{fig:portal-solicitud-impacto}.

\subsection{Gestionar solicitudes desde el portal}

El equipo TI selecciona la pestaña \textit{Gestión del equipo TI}, introduce el token custodiado en Key Vault y carga la bandeja. La vista muestra totales, solicitudes abiertas, trabajo en proceso y casos con SLA vencido. Los filtros permiten acotar por estado, prioridad y tipo.

Al seleccionar una fila se abre el detalle con solicitante, descripción, clasificación, recomendación y fecha objetivo. El operador puede cambiar el estado, la prioridad o el responsable y añadir una nota. Cada actuación queda incorporada al historial; el token se mantiene únicamente en memoria durante la sesión del navegador y no se guarda en almacenamiento local.

Para una evaluación repetible, el botón \textit{Cargar demostración} prepara cinco solicitudes identificadas como datos de prueba: una pendiente de aprobación, una con SLA vencido, una en proceso, una cerrada y una escalada. La operación es idempotente, por lo que volver a pulsarla no duplica los casos existentes. La figura \ref{fig:portal-demo-bandeja} muestra la bandeja resultante.

\subsection{Aprobar o escalar una solicitud sensible}

Las solicitudes sensibles permanecen en estado \texttt{pendiente\_aprobacion} hasta que el responsable indicado toma una decisión. El equipo TI comprueba servicio, activo, entorno e impacto y utiliza los botones \textit{Aprobar} o \textit{Rechazar}. También puede escalar el caso si aumenta su impacto o se acerca el vencimiento del SLA. La decisión conserva actor, fecha y motivo en el historial, como muestra la figura \ref{fig:portal-aprobacion-registrada}.

\subsection{Crear solicitud desde API}

\begin{verbatim}
POST /solicitudes
Content-Type: application/json

{
  "tipo_solicitud": "acceso",
  "titulo": "Acceso VPN",
  "descripcion": "Necesito acceso VPN al entorno cloud",
  "reportado_por": "usuario@empresa.com",
  "prioridad": "media",
  "servicio_id": "vpn",
  "activo_id": "gateway-vpn",
  "entorno": "produccion"
}
\end{verbatim}

El sistema clasificará automáticamente la solicitud y devolverá prioridad, categoría, tipo, nivel de confianza y recomendación. Si se proporciona una prioridad válida de forma explícita, esta prevalece sobre la prioridad inferida por las reglas.

\subsection{Consultar solicitudes}

\begin{verbatim}
GET /solicitudes
Authorization: Bearer <token>
Filtros: estado=abierta, prioridad=alta,
         tipo_solicitud=acceso
\end{verbatim}

Si se accede a esta URL directamente desde el navegador sin cabecera de autorización, la API responde \texttt{\{"error":"No autenticado"\}}, que es el comportamiento esperado para proteger el listado.

\subsection{Consultar métricas}

\begin{verbatim}
GET /metricas
Authorization: Bearer <token>
\end{verbatim}

La respuesta incluye agregados por estado, prioridad y tipo, además de solicitudes en plazo, vencidas y cerradas.

\subsection{Valorar una solicitud cerrada}

Una vez cerrada la solicitud, el portal habilita una valoración de uno a cinco puntos y un comentario opcional. Cada solicitud admite una sola respuesta; un segundo intento se rechaza para no alterar la métrica. Las solicitudes activas tampoco pueden valorarse. El resultado se incorpora al historial y a los indicadores del centro operativo, como se observa en la figura \ref{fig:portal-valoracion-recibida}.

\subsection{Crear solicitud mediante Logic App}

La Logic App permite automatizar la entrada de solicitudes desde un sistema externo. El usuario o sistema integrador envía un JSON al trigger HTTP de la Logic App y el flujo invoca internamente \texttt{POST /solicitudes}. La respuesta final mantiene el mismo formato que la API, incluyendo identificador, prioridad, clasificación y recomendación.

Esta opción está pensada para integraciones empresariales, por ejemplo formularios internos, herramientas de soporte o procesos automáticos que necesiten registrar una solicitud sin acceder manualmente al portal.

Si la solicitud creada requiere aprobación, el mismo flujo registra una notificación para el responsable correspondiente antes de responder al sistema llamante. La gestión continúa en la bandeja TI; la Logic App no sustituye al portal, sino que ofrece un canal automático de entrada y evita duplicar las reglas de validación, clasificación y persistencia de la API.

La URL del trigger contiene una firma de acceso y se obtiene desde Azure Portal únicamente al configurar la integración; no se guarda en el repositorio ni se presenta completa en las evidencias. El diseño conceptual del flujo se recoge en el anexo de diseño y la ejecución real se conserva en \texttt{docs/evidencias/}.

\section{Demostración recomendada para el tribunal}

La demostración puede realizarse de forma estable en el siguiente orden:

\begin{enumerate}
    \item Abrir \texttt{/health} y comprobar que el estado es \texttt{ok} y el almacenamiento es \texttt{azure}.
    \item Abrir \texttt{/portal}, registrar una solicitud y mostrar el identificador, la clasificación, el impacto, el responsable y el plazo objetivo devueltos.
    \item Entrar en \textit{Gestión del equipo TI}, introducir el token recuperado de Key Vault y pulsar \textit{Cargar bandeja}. El secreto no debe pronunciarse, copiarse en el guion ni aparecer en la grabación.
    \item Pulsar \textit{Cargar demostración}. Seleccionar sucesivamente los casos pendiente de aprobación, SLA vencido, en proceso, cerrado y escalado para explicar las diferencias.
    \item Aprobar o escalar el caso sensible y enseñar que el actor, la fecha y el motivo aparecen en el historial.
    \item Abrir \textit{Centro operativo} para mostrar alertas SLA, carga por responsable, distribución por servicio y exportación CSV.
    \item Seleccionar el caso cerrado, registrar una valoración y comprobar el nuevo promedio de satisfacción.
    \item Abrir \texttt{/docs} para enseñar el contrato OpenAPI. Si se ejecuta una operación protegida, usar \textit{Authorize} sin dejar el token visible en la captura.
    \item Abrir \texttt{/acerca} y mostrar versión, arquitectura y enlaces públicos a GitHub, Zube y SonarCloud.
    \item Como cierre opcional, invocar la Logic App con un cuerpo JSON válido y localizar la nueva solicitud en la bandeja TI.
\end{enumerate}

La página \texttt{/ayuda} conserva dentro del propio producto una versión abreviada de este recorrido. Si una integración o credencial no estuviera disponible durante la defensa, las capturas de \texttt{docs/evidencias/} permiten acreditar el resultado sin exponer secretos.
