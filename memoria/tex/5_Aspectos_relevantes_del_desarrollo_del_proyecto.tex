\capitulo{5}{Aspectos relevantes del desarrollo del proyecto}

\section{Ciclo de vida del proyecto}

El proyecto se ha organizado en cinco sprints siguiendo una metodología ágil.

\begin{itemize}
    \item Sprint 1: Análisis del problema empresarial, definición de requisitos y arquitectura inicial.
    \item Sprint 2: Provisionamiento de infraestructura Azure y configuración de Terraform.
    \item Sprint 3: Desarrollo de la API Flask, portal web y persistencia en Azure Storage.
    \item Sprint 4: Seguridad con Key Vault, Managed Identity, portal web y despliegue en Azure.
    \item Sprint 5: Pruebas, calidad SonarCloud, documentación y validación del prototipo.
\end{itemize}

\section{Arquitectura del sistema}

La arquitectura implementada sigue un modelo de servicios gestionados en Microsoft Azure. Los componentes principales son:

\begin{itemize}
    \item \textbf{Azure App Service}: hospeda la API Flask y el portal web empresarial en Python 3.11.
    \item \textbf{Azure Storage}: persiste las solicitudes operativas TI en un contenedor Blob privado.
    \item \textbf{Azure Key Vault}: almacena el token de autenticación de la API.
    \item \textbf{Scripts de despliegue}: automatizan la configuración y publicación en Azure mediante Azure CLI.
\end{itemize}

El flujo principal del sistema es el siguiente:

\begin{enumerate}
    \item Un empleado envía una solicitud operativa TI desde el portal web o mediante la API REST.
    \item Azure App Service procesa la petición con Gunicorn.
    \item La API valida los datos y clasifica automáticamente la solicitud.
    \item El sistema genera una recomendación operativa para el equipo de TI.
    \item Los datos se almacenan en Azure Blob Storage.
    \item Los secretos se obtienen desde Azure Key Vault mediante Managed Identity.
    \item El endpoint de métricas expone el estado agregado del sistema.
\end{enumerate}

\section{API REST implementada}

La API desarrollada en \texttt{src/app.py} expone los siguientes endpoints:

\begin{itemize}
    \item \texttt{GET /}: estado del servicio y versión.
    \item \texttt{GET /portal}: portal web para solicitudes operativas TI.
    \item \texttt{GET /health}: comprobación de salud y modo de almacenamiento.
    \item \texttt{POST /solicitudes}: creación de solicitudes con clasificación automática.
    \item \texttt{GET /solicitudes}: listado con filtros por estado, prioridad y tipo.
    \item \texttt{GET /metricas}: métricas agregadas del sistema.
\end{itemize}

Al crear una solicitud, el módulo \texttt{classifier.py} analiza el título y la descripción para asignar tipo de solicitud (\textit{acceso}, \textit{entorno}, \textit{aplicación}, \textit{configuración}, \textit{incidencia}), prioridad (\textit{baja}, \textit{media}, \textit{alta}), categoría (\textit{seguridad}, \textit{infraestructura}, \textit{soporte}, \textit{general}) y una recomendación operativa, devolviendo un valor de confianza asociado.

\section{Componente de inteligencia artificial}

Se ha incorporado un clasificador ligero basado en análisis semántico de palabras clave y reglas de negocio. Este componente cumple el objetivo del TFG de aportar análisis automático de información operativa sin depender de infraestructuras corporativas complejas. El clasificador evalúa términos relacionados con seguridad, infraestructura y soporte para determinar la categoría, prioridad y recomendación de cada solicitud operativa TI.

\section{Automatización del despliegue}

Se ha implementado un proceso de despliegue reproducible mediante \texttt{scripts/deploy-azure.ps1}. Este script:

\begin{enumerate}
    \item Comprueba la sesión de Azure CLI.
    \item Obtiene la configuración de Azure Storage y Key Vault.
    \item Habilita Managed Identity en Azure App Service.
    \item Concede a la aplicación permisos sobre el secreto almacenado en Key Vault.
    \item Configura variables de entorno y publica el código Python en Azure App Service.
\end{enumerate}

\section{Seguridad y gestión de secretos}

La autenticación de la API utiliza un token Bearer almacenado en Azure Key Vault. La App Service accede al secreto mediante Managed Identity, evitando credenciales embebidas en el código fuente. El almacenamiento de solicitudes reside en un contenedor Blob con acceso privado.

\section{Validación del prototipo}

Las pruebas unitarias en \texttt{tests/test\_api.py} verifican la creación de solicitudes, el listado, las métricas, el portal web y la clasificación automática. El prototipo puede ejecutarse localmente con almacenamiento en fichero JSON o desplegarse en Azure con persistencia en Blob Storage.
