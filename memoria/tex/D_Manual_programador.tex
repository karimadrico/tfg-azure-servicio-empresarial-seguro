\apendice{Documentación técnica de programación}

\section{Introducción}

Este anexo describe la estructura técnica del repositorio y los pasos para ejecutar, probar y desplegar la solución, apoyándose en scripts reproducibles con Azure CLI y servicios gestionados de Azure \cite{azure_keyvault,azure_appservice_keyvault_python}.

\section{Estructura de directorios}

\begin{itemize}
    \item \texttt{src/app.py}: API principal Flask.
    \item \texttt{src/classifier.py}: clasificador automático de incidencias.
    \item \texttt{src/storage.py}: persistencia local, Azure Blob Storage o Azure Cosmos DB.
    \item \texttt{src/config.py}: configuración por variables de entorno.
    \item \texttt{src/catalog.py}: catálogo de servicios, activos y responsables.
    \item \texttt{src/request\_workflow.py}: reglas de impacto, aprobación y SLA.
    \item \texttt{src/operations.py}: métricas operativas, exportación y valoraciones.
    \item \texttt{src/static/openapi.json}: contrato OpenAPI 3.0 de la API.
    \item \texttt{src/requirements.txt}: dependencias Python.
    \item \texttt{tests/test\_api.py}: pruebas unitarias.
    \item \texttt{logicapp/}: definición del flujo de automatización HTTP.
    \item \texttt{infra/terraform/}: infraestructura como código.
    \item \texttt{scripts/}: despliegue y verificación en Azure.
    \item \texttt{memoria/tex/}: memoria y anexos en LaTeX.
\end{itemize}

\section{Ejecución local}

\begin{verbatim}
cd src
pip install -r requirements.txt
set STORAGE_MODE=local
python app.py
\end{verbatim}

La API quedará disponible en \texttt{http://localhost:5000}.

\section{Variables de entorno}

\begin{itemize}
    \item \texttt{STORAGE\_MODE}: \texttt{local} o \texttt{azure}.
    \item \texttt{AZURE\_STORAGE\_CONNECTION\_STRING}: cadena de conexión del Storage Account.
    \item \texttt{KEY\_VAULT\_URL}: URI del Key Vault.
    \item \texttt{API\_KEY}: token Bearer opcional para desarrollo local.
\end{itemize}


\section{Infraestructura como código}

El repositorio incluye definiciones de infraestructura como código en Terraform y Bicep. En el alcance final del TFG, el despliegue operativo se realiza mediante scripts PowerShell y Azure CLI, mientras que Terraform y Bicep actúan como documentación técnica reproducible de los recursos cloud necesarios. Esta separación permite defender tanto la ejecución práctica del despliegue como el diseño declarativo de la infraestructura.

\section{Despliegue en Azure}

\begin{enumerate}
    \item Iniciar sesión con \texttt{az login}.
    \item Definir la variable de entorno \texttt{API\_KEY} con el token que se almacenará en Key Vault.
    \item Ejecutar \texttt{scripts/deploy-azure.ps1} desde la raíz del repositorio.
    \item Comprobar \texttt{/health} y abrir el portal web desplegado.
    \item Ejecutar \texttt{scripts/verify-azure.ps1} para verificar endpoints principales.
    \item Ejecutar \texttt{scripts/deploy-logicapp.ps1} para publicar el flujo HTTP de automatización cuando se desee probar la integración externa.
\end{enumerate}

\section{Mantenimiento operativo}

El mantenimiento del prototipo se organiza alrededor de los siguientes puntos:

\begin{itemize}
    \item Revisar el endpoint \texttt{/health} tras cada despliegue.
    \item Ejecutar \texttt{scripts/verify-azure.ps1} para comprobar portal, API, creación de solicitudes, listado y métricas.
    \item Mantener la clave de API en Azure Key Vault, evitando valores embebidos en el repositorio.
    \item Revisar Application Insights y Azure Monitor para detectar errores, tiempos de respuesta o ausencia de peticiones.
    \item Revisar SonarCloud tras cambios relevantes para mantener el Quality Gate aprobado.
    \item Documentar evidencias finales en \texttt{docs/evidencias/} antes de generar los entregables.
\end{itemize}
