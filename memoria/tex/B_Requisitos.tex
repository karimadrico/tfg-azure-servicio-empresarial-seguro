\section{Introducción}

Este anexo describe los requisitos funcionales y no funcionales definidos para el sistema desarrollado.

\section{Objetivos generales}

El objetivo del sistema es permitir la gestión segura de solicitudes operativas TI utilizando servicios cloud desplegados en Microsoft Azure. Una incidencia se considera un tipo concreto de solicitud, junto con accesos, cambios de configuración, altas de aplicaciones o creación de entornos.

\section{Especificación de requisitos}

\subsection{RF-01 Crear solicitudes operativas TI}
\begin{sloppypar}
El sistema debe permitir registrar solicitudes mediante \texttt{POST /solicitudes}, validando título, descripción, tipo de solicitud y email del solicitante. El endpoint \texttt{POST /incidencias} se mantiene como compatibilidad para el caso específico de incidencias.
\end{sloppypar}

\subsection{RF-02 Consultar solicitudes}
El sistema debe listar solicitudes mediante \texttt{GET /solicitudes}, con filtros opcionales por estado, prioridad y tipo. Esta operación requiere autenticación mediante token Bearer.

\subsection{RF-03 Obtener métricas}
El sistema debe exponer métricas agregadas en \texttt{GET /metricas}.

\subsection{RF-04 Gestionar almacenamiento cloud}
Las solicitudes deben persistirse en Azure Blob Storage en entorno cloud y en fichero JSON en desarrollo local.

\subsection{RF-05 Gestionar secretos mediante Azure Key Vault}
El token de autenticación debe almacenarse en Key Vault y recuperarse mediante Managed Identity.

\subsection{RF-06 Clasificación automática}
El sistema debe clasificar automáticamente prioridad, categoría y tipo de cada solicitud, generando además una recomendación operativa.

\subsection{RF-07 Automatización de entrada mediante Logic App}
El sistema debe permitir que un sistema externo cree solicitudes mediante una Logic App con trigger HTTP, manteniendo la validación, clasificación y persistencia centralizadas en la API.

\subsection{RF-08 Gestión del ciclo de vida}
El equipo TI debe poder consultar una solicitud concreta, asignar un responsable y actualizar su estado o prioridad. Las transiciones se validan para impedir estados desconocidos y permitir la reapertura controlada de solicitudes cerradas.

\subsection{RF-09 Trazabilidad y plazo objetivo}
Cada solicitud debe conservar la fecha de creación y actualización, un objetivo de atención calculado según su prioridad y un historial de actuaciones con fecha, actor y detalle del cambio.

\subsection{RF-10 Catálogo de servicios y activos}
El portal y la API deben permitir seleccionar un servicio empresarial, uno de sus activos y un entorno admitido. La API rechaza combinaciones que no pertenezcan al catálogo para evitar registros ambiguos.

\subsection{RF-11 Cálculo de impacto y propiedad}
El sistema debe combinar criticidad del servicio, activo, entorno y prioridad para obtener un impacto operativo. El propietario definido en el catálogo se utiliza como responsable inicial de la solicitud.

\subsection{RF-12 Aprobación de solicitudes sensibles}
Las solicitudes cuyo tipo y contexto se consideren sensibles deben permanecer pendientes hasta que el aprobador acepte o rechace la actuación. La Logic App registra la notificación y la API conserva la decisión en el historial.

\subsection{RF-13 Centro operativo y escalado}
El equipo TI debe poder escalar solicitudes y consultar aprobaciones pendientes, carga por responsable, distribución por servicio, impacto crítico, tiempos de resolución y alertas de SLA.

\subsection{RF-14 Exportación de informe}
El sistema debe generar un informe CSV de las solicitudes para su revisión externa. Los valores se neutralizan cuando comienzan por caracteres interpretables como fórmulas por una hoja de cálculo.

\subsection{RF-15 Escenario de demostración}
El equipo evaluador debe poder cargar de forma controlada cinco casos representativos: aprobación pendiente, SLA vencido, solicitud en proceso, solicitud cerrada y caso escalado. La operación debe ser idempotente para no duplicar los registros al repetir la demostración.

\subsection{RF-16 Ayuda y contrato de la API}
El sistema debe ofrecer una guía de uso, información verificable de versión y arquitectura, y un contrato OpenAPI 3.0 navegable. Las operaciones protegidas deben poder probarse desde Swagger UI mediante un token Bearer obtenido de Key Vault.

\subsection{RF-17 Valoración del servicio}
Una solicitud cerrada debe admitir una única valoración de uno a cinco puntos y un comentario opcional. El centro operativo debe agregar el número de respuestas y la satisfacción media sin permitir valoraciones sobre solicitudes todavía activas.

\subsection{RNF-01 Seguridad}
Autenticación Bearer, almacenamiento privado y gestión de secretos centralizada.

\subsection{RNF-02 Escalabilidad}
Arquitectura basada en servicios gestionados de Azure App Service y Blob Storage.

\subsection{RNF-03 Disponibilidad}
Endpoint de salud \texttt{/health} y despliegue automatizado.

\subsection{RNF-04 Trazabilidad}
Registro de identificador, fechas, estado, prioridad, responsable, clasificación e historial de cada solicitud.

\subsection{RNF-05 Observabilidad}
El sistema debe aportar evidencias de monitorización mediante métricas de la API, Application Insights y Azure Monitor, de forma que pueda analizarse el comportamiento del prototipo desplegado.
