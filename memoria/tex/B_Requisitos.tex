\apendice{Especificación de Requisitos}

\section{Introducción}

Este anexo describe los requisitos funcionales y no funcionales definidos para el sistema desarrollado.

\section{Objetivos generales}

El objetivo del sistema es permitir la gestión segura de solicitudes operativas TI utilizando servicios cloud desplegados en Microsoft Azure. Una incidencia se considera un tipo concreto de solicitud, junto con accesos, cambios de configuración, altas de aplicaciones o creación de entornos.

\section{Catálogo de requisitos}

\subsection{Requisitos funcionales}

\begin{itemize}
    \item RF-01 Crear solicitudes operativas TI.
    \item RF-02 Consultar solicitudes.
    \item RF-03 Obtener métricas del sistema.
    \item RF-04 Gestionar almacenamiento cloud.
    \item RF-05 Gestionar secretos mediante Azure Key Vault.
    \item RF-06 Clasificar solicitudes automáticamente.
\end{itemize}

\subsection{Requisitos no funcionales}

\begin{itemize}
    \item RNF-01 Seguridad.
    \item RNF-02 Escalabilidad.
    \item RNF-03 Disponibilidad.
    \item RNF-04 Trazabilidad.
\end{itemize}

\section{Especificación de requisitos}

\subsection{RF-01 Crear solicitudes operativas TI}
El sistema debe permitir registrar solicitudes mediante \texttt{POST /solicitudes}, validando título, descripción, tipo de solicitud y email del solicitante. El endpoint \texttt{POST /incidencias} se mantiene como compatibilidad para el caso específico de incidencias.

\subsection{RF-02 Consultar solicitudes}
El sistema debe listar solicitudes mediante \texttt{GET /solicitudes}, con filtros opcionales por estado, prioridad y tipo. Esta operación requiere autenticación mediante token Bearer.

\subsection{RF-03 Obtener métricas}
El sistema debe exponer métricas agregadas en \texttt{GET /metricas}.

\subsection{RF-04 Gestionar almacenamiento cloud}
Las solicitudes deben persistirse en Azure Blob Storage en entorno cloud y en fichero JSON en desarrollo local.

\subsection{RF-05 Gestionar secretos mediante Azure Key Vault}
El token de autenticación debe almacenarse en Key Vault y recuperarse mediante Managed Identity.

\subsection{RF-06 Clasificación automática}
El sistema debe clasificar automáticamente prioridad, categoría y tipo de cada solicitud, generando además una recomendación operativa.

\subsection{RNF-01 Seguridad}
Autenticación Bearer, almacenamiento privado y gestión de secretos centralizada.

\subsection{RNF-02 Escalabilidad}
Arquitectura basada en servicios gestionados de Azure App Service y Blob Storage.

\subsection{RNF-03 Disponibilidad}
Endpoint de salud \texttt{/health} y despliegue automatizado.

\subsection{RNF-04 Trazabilidad}
Registro de fecha de creación, estado, prioridad, tipo, clasificación y recomendación de cada solicitud.
