\section{Introducción}

Este anexo describe el diseño general de la plataforma cloud para la automatización de solicitudes operativas TI.

\section{Diseño arquitectónico}

La arquitectura sigue un modelo basado en servicios cloud desplegados sobre Microsoft Azure. La aplicación se ejecuta en Azure App Service, utiliza Azure Blob Storage para persistir solicitudes, Azure Key Vault para secretos y Managed Identity para reducir la exposición de credenciales \cite{azure_appservice_python,azure_blob_python,azure_appservice_managed_identity,azure_appservice_keyvault_python}. Además, incorpora Azure Logic Apps como mecanismo de automatización de entrada y Application Insights/Azure Monitor para observabilidad básica del servicio \cite{azure_logic_apps,azure_monitor}.

Los principales componentes son:

\begin{itemize}
    \item Azure App Service (API Flask y portal web).
    \item Azure Storage (persistencia de solicitudes).
    \item Azure Key Vault (gestión de secretos).
    \item Azure Logic Apps (automatización HTTP de solicitudes).
    \item Application Insights y Azure Monitor (observabilidad del despliegue).
    \item Script PowerShell con Azure CLI para despliegue reproducible.
\end{itemize}

\section{Diseño procedimental}

El flujo principal del sistema es:

\begin{enumerate}
    \item Recepción de solicitudes operativas TI por portal web o API REST.
    \item Validación de campos obligatorios.
    \item Clasificación automática mediante componente de IA ligera.
    \item Generación de recomendaciones de actuación operativa.
    \item Persistencia de solicitudes en Blob Storage o fichero local.
    \item Respuesta JSON con identificador, prioridad, tipo, recomendación y confianza.
    \item Verificación del despliegue mediante endpoint \texttt{/health}, portal web y pruebas HTTP.
\end{enumerate}

\subsection{Tipos de solicitud soportados}

\begin{itemize}
    \item Acceso a recursos (VPN, permisos, cuentas).
    \item Provisión de entornos (desarrollo, staging, producción).
    \item Alta de aplicaciones o servicios.
    \item Cambios de configuración.
    \item Incidencias o peticiones de soporte.
\end{itemize}

\section{Integración cloud}

\begin{itemize}
    \item \textbf{Terraform}: documenta Resource Group, App Service, Storage y Key Vault como infraestructura declarativa.
    \item \textbf{Bicep}: plantillas alternativas para reproducir la infraestructura.
    \item \textbf{Script PowerShell}: configura recursos existentes y publica la aplicación en Azure App Service.
    \item \textbf{Logic App}: expone un trigger HTTP que transforma una solicitud externa en una llamada controlada a \texttt{POST /solicitudes}.
    \item \textbf{Application Insights}: recoge telemetría del App Service para apoyar la supervisión del prototipo.
\end{itemize}

\section{Automatización con Logic App}

La Logic App añadida al prototipo actúa como punto de entrada para sistemas externos que necesiten generar solicitudes sin utilizar directamente el portal web. El flujo recibe una petición HTTP con los campos principales de la solicitud, invoca la API desplegada en App Service mediante \texttt{POST /solicitudes} y devuelve al llamante la respuesta generada por la propia API. De esta forma, la lógica de validación, clasificación automática y persistencia se mantiene centralizada en el backend.

Esta decisión permite defender un caso de automatización empresarial realista: un formulario externo, una herramienta de soporte o un proceso corporativo podrían enviar solicitudes al flujo HTTP y delegar en la plataforma cloud la clasificación, priorización y registro de la petición.

\section{Diseño de datos}

La persistencia del prototipo utiliza un documento JSON por solicitud en Azure Blob Storage. No existe una base de datos relacional, por lo que el modelo de datos se describe como una entidad documental.

\begin{table}[h]
\centering
\small
\begin{tabularx}{\textwidth}{l l X}
\textbf{Campo} & \textbf{Tipo} & \textbf{Descripción} \\
\hline
id & texto & Identificador único de la solicitud. \\
titulo & texto & Resumen de la petición enviada por el usuario. \\
descripcion & texto & Detalle de la solicitud o incidencia. \\
tipo\_solicitud & texto & Acceso, entorno, aplicación, configuración o incidencia. \\
prioridad & texto & Prioridad calculada por el clasificador. \\
categoria & texto & Área funcional detectada: infraestructura, seguridad, software u otra. \\
estado & texto & Estado operativo de la solicitud. \\
recomendacion & texto & Acción sugerida para el equipo TI. \\
reportado\_por & texto & Correo o identificador del solicitante. \\
created\_at & fecha & Fecha de creación de la solicitud. \\
\end{tabularx}

\section{Calidad y observabilidad}

La calidad se comprueba mediante pruebas automáticas y análisis manual en SonarCloud. El panel de SonarCloud se aporta como evidencia de mantenibilidad, fiabilidad, seguridad y duplicidad del código. La observabilidad técnica se apoya en el endpoint \texttt{/health}, en los logs de App Service y en las métricas disponibles desde Application Insights y Azure Monitor \cite{azure_monitor}.
