\apendice{Especificación de diseño}

\section{Introducción}

Este anexo describe el diseño general de la plataforma cloud para la automatización de solicitudes operativas TI.

\section{Diseño arquitectónico}

La arquitectura sigue un modelo basado en servicios cloud desplegados sobre Microsoft Azure. La aplicación se ejecuta en Azure App Service, utiliza Azure Blob Storage para persistir solicitudes, Azure Key Vault para secretos y Managed Identity para reducir la exposición de credenciales \cite{azure_appservice_python,azure_blob_python,azure_appservice_managed_identity,azure_appservice_keyvault_refs}.

Los principales componentes son:

\begin{itemize}
    \item Azure App Service (API Flask y portal web).
    \item Azure Storage (persistencia de solicitudes).
    \item Azure Key Vault (gestión de secretos).
    \item Script PowerShell con Azure CLI para despliegue reproducible.
\end{itemize}

\section{Diseño procedimental}

El flujo principal del sistema es:

\begin{enumerate}
    \item Recepción de solicitudes operativas TI por portal web o API REST.
    \item Validación de campos obligatorios.
    \item Clasificación automática mediante componente de IA ligera.
    \item Generación de recomendaciones de actuación operativa.
    \item Persistencia de solicitudes en Blob Storage o fichero local.
    \item Respuesta JSON con identificador, prioridad, tipo, recomendación y confianza.
    \item Verificación del despliegue mediante endpoint \texttt{/health}, portal web y pruebas HTTP.
\end{enumerate}

\subsection{Tipos de solicitud soportados}

\begin{itemize}
    \item Acceso a recursos (VPN, permisos, cuentas).
    \item Provisión de entornos (desarrollo, staging, producción).
    \item Alta de aplicaciones o servicios.
    \item Cambios de configuración.
    \item Incidencias o peticiones de soporte.
\end{itemize}

\section{Integración cloud}

\begin{itemize}
    \item \textbf{Terraform}: documenta Resource Group, App Service, Storage y Key Vault como infraestructura declarativa.
    \item \textbf{Bicep}: plantillas alternativas para reproducir la infraestructura.
    \item \textbf{Script PowerShell}: configura recursos existentes y publica la aplicación en Azure App Service.
\end{itemize}

\section{Diseño de datos}

\section{Calidad y observabilidad}

La calidad se comprueba mediante pruebas automáticas y análisis manual en SonarCloud. El panel de SonarCloud se aporta como evidencia de mantenibilidad, fiabilidad, seguridad y duplicidad del código. La observabilidad técnica se apoya en el endpoint \texttt{/health}, en los logs de App Service y en las métricas disponibles desde Azure Monitor \cite{azure_monitor}.
