\section{Diseño arquitectónico}

La arquitectura sigue un modelo basado en servicios cloud desplegados sobre Microsoft Azure. La aplicación se ejecuta en Azure App Service, utiliza Azure Cosmos DB para persistir solicitudes, Azure Key Vault para secretos y Managed Identity para reducir la exposición de credenciales \cite{azure_appservice_python,azure_cosmos_python,azure_appservice_managed_identity,azure_appservice_keyvault_python}. Además, incorpora Azure Logic Apps como mecanismo de automatización de entrada y Application Insights/Azure Monitor para observabilidad básica del servicio \cite{azure_logic_apps,azure_monitor}.

Los principales componentes son:

\begin{itemize}
    \item Azure App Service (API Flask y portal web).
    \item Azure Cosmos DB (persistencia documental de solicitudes).
    \item Azure Key Vault (gestión de secretos).
    \item Azure Logic Apps (automatización HTTP de solicitudes).
    \item Application Insights y Azure Monitor (observabilidad del despliegue).
    \item Script PowerShell con Azure CLI para despliegue reproducible.
\end{itemize}

\section{Diseño procedimental}

El flujo principal del sistema es:

\begin{enumerate}
    \item Recepción de solicitudes operativas TI por portal, API REST o Logic App.
    \item Validación de campos obligatorios y del contexto formado por servicio, activo y entorno.
    \item Clasificación automática mediante reglas de negocio y cálculo del impacto empresarial.
    \item Asignación del equipo propietario y del objetivo SLA.
    \item Bloqueo en estado pendiente cuando la criticidad exige aprobación.
    \item Persistencia de la solicitud y de su historial en Azure Cosmos DB.
    \item Gestión posterior mediante aprobación, rechazo, escalado, asignación y transiciones de estado.
    \item Supervisión de carga, vencimientos y alertas desde el centro operativo.
    \item Cierre de la solicitud y valoración opcional del servicio por el solicitante.
    \item Exportación del informe CSV y verificación mediante pruebas HTTP, escenario de demostración y contrato OpenAPI.
\end{enumerate}

La secuencia completa de validación y persistencia se detalla en la figura \ref{fig:secuencia-persistencia}, dentro del diseño de datos.

\begin{itemize}
    \item \textbf{Terraform}: documenta Resource Group, App Service, Storage, Key Vault, RBAC y Application Insights como infraestructura declarativa.
    \item \textbf{Bicep}: plantillas alternativas para reproducir la infraestructura.
    \item \textbf{Script PowerShell}: configura recursos existentes y publica la aplicación en Azure App Service.
    \item \textbf{Logic App}: expone un trigger HTTP que transforma una solicitud externa en una llamada controlada a \texttt{POST /solicitudes}.

\end{itemize}

\section{Automatización con Logic App}

La Logic App actúa como punto de entrada para sistemas externos que no utilizan el portal. Recibe una petición HTTP, llama a \texttt{POST /solicitudes} en App Service y devuelve la respuesta de la API. La validación, las reglas y la persistencia siguen en el backend; el flujo solo orquesta la entrada y no duplica lógica de negocio.

Esta decisión permite defender un caso de automatización empresarial realista: un formulario externo, una herramienta de soporte, un buzón corporativo o un proceso interno podrían enviar solicitudes al flujo HTTP y delegar en la plataforma cloud la clasificación, priorización y registro de la petición. En una empresa, la Logic App sería mantenida por el equipo de TI o de plataforma cloud, mientras que los usuarios internos consumirían el portal o los formularios integrados. La gestión operativa seguiría centralizada en la API y en las vistas de consulta, evitando duplicar reglas de negocio en cada canal de entrada.

\section{Diseño de datos}

Los datos son semiestructurados y se serializan como JSON. En Azure, cada solicitud se almacena como documento independiente en Cosmos DB; en ejecución local se utiliza un fichero JSON con el mismo formato funcional para facilitar las pruebas. La API es la única responsable de leer, validar y modificar los documentos. El portal consume la API y la Logic App envía peticiones HTTP, pero ninguno de los dos accede directamente a la base de datos. Esta frontera evita duplicar reglas entre canales y permite reutilizar el mismo modelo local y cloud.

El historial se almacena dentro de cada solicitud y se inicializa con un evento de creación. Por tanto, toda solicitud válida incorpora al menos un evento y puede acumular muchos más durante aprobaciones, escalados, cambios de estado, cierre o valoración. Cada evento contiene fecha, acción, actor y detalle, lo que permite reconstruir la evolución sin consultar una estructura externa y materializa los requisitos RF-09 y RNF-04.

\textbf{Regla} & \textbf{Comprobación o cálculo} & \textbf{Efecto} \\
\hline

\subsection{Concurrencia y evolución}

La implementación final utiliza Cosmos DB y guarda cada solicitud como documento independiente mediante operaciones de inserción o actualización. Este cambio elimina la dependencia del blob único y reduce el riesgo de sobrescribir la colección completa. No obstante, el prototipo no implementa aún control de concurrencia optimista con ETag, condición \texttt{If-Match}, transacciones ni bloqueo distribuido. En una carga productiva, dos operaciones simultáneas sobre la misma solicitud deberían coordinarse con esas técnicas para evitar escrituras perdidas o decisiones contradictorias.

La versión inicial con Blob Storage era suficiente para la demostración de bajo volumen, pero no para una plataforma ITSM con concurrencia real. La evolución a Cosmos DB separa cada solicitud en un documento propio y reduce el acoplamiento del blob único. Aun así, una implantación productiva debería reforzar el diseño con control de concurrencia optimista, particionado revisado, identidad individual, auditoría y políticas de retención. El análisis de esta evolución se conserva en \path{docs/decisiones/concurrencia-blob.md}.
